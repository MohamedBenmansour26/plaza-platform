## PLZ-006 ‚Äî Supabase schema + RLS ‚Äî 06 April 2026

**What I built:** Full Postgres schema in a single migration (`supabase/migrations/20260406000000_initial_schema.sql`): 5 tables (merchants, products, orders, order_items, deliveries), 5 enums, FK constraints, check constraints, indexes, and RLS policies. TypeScript types in `types/supabase.ts` and both Supabase clients typed with the `Database` generic.

**Decisions made:**
- Single migration file for all tables ‚Äî cleaner at initial schema stage; incremental migrations make more sense once the DB is live and has data
- `order_items.unit_price` is a snapshot field, not an FK to `products.price` ‚Äî this prevents historical orders from being affected when a merchant changes a product's price; important for billing correctness
- `deliveries.fee` uses a `check (fee in (2900, 3900))` constraint ‚Äî enforces Plaza's flat pricing at the DB level, not just application level
- `merchants` has a unique constraint on `user_id` (one store per user at MVP) ‚Äî enforced at DB level
- `deliveries` has a unique constraint on `order_id` (one delivery per order) ‚Äî prevents duplicate dispatches
- RLS public INSERT on `orders` and `order_items` uses `with check (true)` ‚Äî anonymous customers can place orders without an account; correct for MVP storefront flow
- Types written by hand (not generated by `supabase gen types`) ‚Äî Supabase CLI requires Docker for local type generation, which is not available in this environment. Types manually verified against the migration SQL.
- Both `createBrowserClient` and `createServerClient` now accept `Database` generic ‚Äî enables type-safe `.from('merchants')` queries throughout the app

**Shortcuts taken:**
- TypeScript types hand-written, not generated ‚Äî they should be regenerated from a running local Supabase instance once Docker is available, or via `supabase gen types typescript --project-id <id>` against the remote project. Flagged in PR notes for Anas.
- No E2E test written for this task ‚Äî there is no UI to test (schema-only). RLS policies need manual SQL verification against a running Supabase instance.

**Friction encountered:**
- Local repo had no `main` branch tracked ‚Äî had to `git fetch origin` to pull it down after confirming PLZ-003 PR was merged on GitHub
- `supabase gen types typescript --local` not available (no Docker) ‚Äî had to hand-write types
- `git push` via background task did not produce output; had to embed token in remote URL directly

**What I would do differently:**
- Set up Docker locally so Supabase CLI can run migrations and generate types properly; note this as a prerequisite in CLAUDE.md
- Add a `package.json` script `"db:types": "supabase gen types typescript --local > types/supabase.ts"` so future migrations have a reproducible type generation step

## PLZ-008 ‚Äî Merchant onboarding flow ‚Äî 06 April 2026

**What I built:** 2-step onboarding wizard at `/onboarding`: step 1 (store name, auto-slug with debounced uniqueness check, description, optional logo upload), step 2 (bilingual product name FR+AR, price in MAD stored as centimes, stock, required photo upload). Server actions for slug check + submit. Upload API at `/api/upload`. `Textarea` UI primitive. `onboarding` i18n namespace in both languages. Playwright E2E test.

**Decisions made:**
- Split client form into `OnboardingForm` (orchestrator) + `StepStoreDetails` + `StepFirstProduct` ‚Äî keeps each file under 150 lines, clean props interface
- Upload happens immediately on file selection (not on submit) ‚Äî better UX; URL stored in state and included in final submit payload
- `submitOnboardingAction` uses service client for upsert ‚Äî RLS only allows `user_id = auth.uid()` reads/writes on the merchant row; server actions need elevated access
- Price input: user types whole MAD (e.g. 150), stored as centimes (15000) via `Math.round(parseFloat(price) * 100)`
- Slug debounce: 500ms via `useRef + setTimeout` (no lodash); slug status shown inline below the input
- `maybeSingle<Pick<Merchant, 'id' | 'store_slug'>>()` ‚Äî explicit generic bypasses Supabase JS 2.x / TS 5.9 inferred-result regression where column-select results resolve to `never`
- Middleware updated to add `/onboarding` to `PROTECTED_PREFIXES` ‚Äî unauthenticated redirect handled at edge level

**Shortcuts taken:**
- No Supabase Storage buckets created ‚Äî `merchant-logos` and `product-images` buckets must be created in the Supabase dashboard before testing uploads. Flagged in PR notes.
- Upload API returns 500 if bucket doesn't exist ‚Äî no auto-create logic; would be a privilege escalation risk
- E2E test uses in-memory pixel PNG buffer for file upload ‚Äî avoids fixture file; `setInputFiles()` with buffer works in Playwright

**What I would do differently:**

## Standing rules

- **i18n wiring:** Always call `useTranslations` in the consuming component. Adding keys to `fr.json`/`ar.json` does nothing if the component never calls `t()`. Before opening any PR, verify every client component with user-facing strings has `const t = useTranslations(...)` and uses it.
- **i18n ownership:** Coordinate with Hamza before touching `messages/fr.json` or `messages/ar.json`. One person owns i18n per sprint.
- **Supabase `.maybeSingle()` / `.single()`:** Always use explicit generic: `.maybeSingle<Pick<T, 'col1' | 'col2'>>()`. Never rely on inference.
- **Supabase `.update()`:** Cast argument `as never` when TS2345 fires ‚Äî `any` is no longer assignable to `never` in TS 5.x strict mode.
